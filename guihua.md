# 规划调用流程

## 前置输入要求（规划提供）

1.  规划输出JSON：
    ```json
    [
      {
        "stepNo": 1,
        "funcType": "功能名",
        "funcNames": ["函数1", "函数2"],
        "description": "该步骤的自然语言描述"
      }
    ]
    ```

2.  字段规则：
    | 字段名 | 类型 | 规则说明 |
    |--------|------|----------|
    | stepNo | 整数 | 从1开始的连续自然数，不能跳号、重复 |
    | funcType | 字符串 | 不能为空，必须是规范内定义的合法功能名 |
    | funcNames | 数组 | 无函数时使用空数组`[]`，多个函数以数组元素存储 |
    | description | 字符串 | 不能为空，必须明确描述操作对象、操作方式和结果 |

## 整体执行流程（串行）

输入校验 → 逐行执行（单步功能规范化 → 单步函数规范化）→ 合并输出完整NLC

## 步骤1：输入校验

1.  校验JSON格式合法性，解析为步骤数组
2.  校验步骤数组非空，且所有`stepNo`连续递增
3.  校验每个步骤的`funcType`非空、`description`非空
4.  校验通过进入下一步，失败返回：`0; 错误，输入格式错误：<具体原因>`

## 步骤2：逐行处理（按stepNo顺序，每行完整处理：功能规范化 → 函数规范化）

### 子步骤2.1：单步功能规范化（当前行）
1.  拼接文件：功能_公共.md
2.  根据当前步骤的`funcType`，找到"功能"目录下的md文件。拼接该文件
3.  拼接上下文环境（各部分之间换行）：
    3.1 拼接焦点表，格式："焦点表=<具体焦点表名>({<字段名>|,})"。其中，主键字段加星号来表示，{}表示可能存在多个同类型的内容。比如：焦点表=XX例子表(*订单ID, 订单日期, 订单金额)。或者：焦点表=课程例子表(班级, 课程名称, 上课日期, 上课时间, 教师)。
    3.2 拼接焦点表之外的引用表，格式："引用表={<具体引用表名>({<字段名>|, })|, }"。其中，主键字段加星号来表示。比如：引用表=成绩例子表(*学生ID, *科目ID, 成绩), 产品例子表(产品ID, 产品厂家, 产品数量), Orders例子表(Order编号,Client编号,金额)
    3.3 拼接非表的引用变量，格式：引用变量={<参数名>=<参数值>|,}。 比如：引用变量=指定部门=[销售部,人力资源],时间参数=2019-01-02,指定金额=3000
4.  拼接文字：该步骤的自然语言描述
5.  将完整文本发送到LLM
6.  解析输出：
    - 成功：提取`1; <功能规范的NLC>`
    - 失败：返回`0; 错误，步骤<序号>功能规范化失败：<错误内容>`，终止流程

### 子步骤2.2：单步函数规范化（当前行）
1.  若当前步骤的`funcNames`数组为空，直接复用步骤2.1的功能规范NLC片段，跳过本步骤
2.  拼接文件：函数_公共.md
3.  根据当前步骤的`funcNames`数组，找到"函数"目录下的md文件。拼接所有找到的文件。
4.  拼接第二步正确的输出结果，即`<功能规范的NLC>`
5.  将完整文本发送到LLM。（如果正确，会返回`1; <规范的NLC>`）
6.  解析输出：
    - 成功：提取`1; <规范的NLC>`中的最终NLC片段
    - 失败：返回`0; 错误，步骤<序号>函数规范化失败：<错误内容>`，终止流程

## 步骤3：合并输出

1.  按`stepNo`顺序拼接所有行已处理完成的最终NLC片段，每行一条
2.  最终成功输出格式：
    ```json
    {
      "code": 1,
      "nlcScript": [
        "步骤1规范NLC",
        "步骤2规范NLC",
        "...",
        "步骤n规范NLC"
      ],
      "rawOutput": "1;\n步骤1规范NLC\n步骤2规范NLC\n...\n步骤n规范NLC"
    }
    ```
